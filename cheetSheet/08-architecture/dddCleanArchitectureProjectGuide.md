# DDD × クリーンアーキテクチャ 設計・実装ガイド

> DDD とクリーンアーキテクチャは、どちらも「コードをきれいに分けるため」だけの考え方ではありません。
> 本質は、業務ルールを正しく理解し、変更に強い形で実装へ落とすことです。

## DDD とクリーンアーキテクチャの役割

まず、DDD とクリーンアーキテクチャは担当する関心が少し違います。

| 考え方                 | 主な関心                                 | 簡単にいうと                             |
| ---------------------- | ---------------------------------------- | ---------------------------------------- |
| DDD                    | 業務をどう理解してモデル化するか         | 業務ルールをコードで表現するための考え方 |
| クリーンアーキテクチャ | コードをどう分離して依存関係を整理するか | 業務ルールを技術都合から守るための構造   |

DDD は、

> この業務では、何が重要な概念なのか？  
> どんなルールで状態が変わるのか？

を考えるためのものです。

クリーンアーキテクチャは、

> その業務ルールを、DB・画面・API・外部サービスに引っ張られない形でどこに置くか？

を考えるためのものです。

つまり、ざっくりいうと以下です。

```text
DDD
  ↓
業務ルールを見つける・整理する

クリーンアーキテクチャ
  ↓
整理した業務ルールを守れる場所に配置する
```

## 基本設計書で守るべき視点

DDD とクリーンアーキテクチャを使う場合、基本設計書では以下を意識します。

### 1. 画面や DB から書き始めない

いきなり、

- 画面項目
- API 項目
- DB カラム
- リクエストパラメータ

から考え始めると、業務ルールが見えにくくなります。

もちろん最終的には画面・API・DB も必要です。  
しかし、最初に考えるべきなのは **業務として何が起きるのか** です。

悪い例です。

```text
注文キャンセルAPIを作る。
リクエストで orderId を受け取り、orders テーブルの status を CANCELED に更新する。
```

一見よさそうですが、これだと業務ルールがわかりません。

- どの注文でもキャンセルできるのか
- 出荷済みでもキャンセルできるのか
- 支払い済みの場合は返金が必要なのか
- キャンセル時に在庫を戻すのか
- キャンセル理由は必要なのか

が読み取れません。

DDD / クリーンアーキテクチャを意識するなら、先にこう書きます。

```text
注文は、未出荷の場合のみキャンセルできる。
注文をキャンセルすると、注文ステータスは「キャンセル済み」になる。
支払い済みの場合は返金処理を行う。
在庫引当済みの場合は在庫を戻す。
```

そのあとで、API や DB に落とします。

```text
注文キャンセルAPIでは orderId と cancelReason を受け取る。
注文情報を取得し、キャンセル可能条件を満たす場合のみキャンセル処理を実行する。
処理結果として、キャンセル後の注文ステータスを返却する。
```

> 設計書では、まず「業務として何が正しいか」を書く。  
> そのあとに「どう実装するか」を書く。

---

### 2. 業務ルールと技術処理を混ぜない

基本設計書では、業務ルールと技術処理を分けて書きます。

| 種類       | 内容                                                           |
| ---------- | -------------------------------------------------------------- |
| 業務ルール | キャンセルできる条件、金額計算、状態遷移、制約など             |
| 技術処理   | API、DB、SQL、HTTP、トランザクション、外部サービス呼び出しなど |

混ぜて書くと、どこが業務仕様で、どこが実装都合なのかわかりにくくなります。

例えばこれは混ざっています。

```text
orders テーブルから status を取得し、status が SHIPPED でなければ CANCELED に更新する。
```

業務ルールとしては、こう書いた方がわかりやすいです。

```text
注文は、出荷済みの場合キャンセルできない。
未出荷の場合のみキャンセルできる。
```

技術処理としては、別でこう書きます。

```text
注文IDをもとに注文情報を取得する。
取得した注文のステータスを確認し、キャンセル可能な場合のみ更新する。
```

> 業務ルールは「業務の言葉」で書く。  
> 技術処理は「実装の言葉」で書く。

---

### 3. 状態が変わる条件を必ず書く

実務で特に重要なのが、状態変更です。

例えば、

- 注文が作成される
- 注文がキャンセルされる
- 支払いが完了する
- 出荷される
- ユーザーが退会する
- 申請が承認される

のような処理では、状態が変わります。

状態が変わる処理では、必ず以下を書きます。

```text
何が
どの状態から
どの条件で
どの状態へ変わるのか
```

例です。

```text
注文は、ステータスが「受付済み」の場合のみキャンセルできる。
キャンセル後、注文ステータスは「キャンセル済み」になる。
ステータスが「出荷済み」の注文はキャンセルできない。
```

状態遷移は、表にすると実装者が理解しやすいです。

| 現在の状態     | 操作       | 次の状態       | 可否 | 理由                       |
| -------------- | ---------- | -------------- | ---- | -------------------------- |
| 受付済み       | キャンセル | キャンセル済み | 可   | まだ出荷されていないため   |
| 出荷準備中     | キャンセル | キャンセル済み | 可   | 出荷前のため               |
| 出荷済み       | キャンセル | -              | 不可 | 商品が発送済みのため       |
| キャンセル済み | キャンセル | -              | 不可 | すでにキャンセル済みのため |

この表があると、実装では Entity や Domain Service に判断ロジックを置きやすくなります。

## 業務ルールを明確にするためにどうするか

「業務ルールを明確にする」と言われても、最初はかなり難しいです。

実務では、以下のように進めると整理しやすいです。

---

### 1. 業務イベントを洗い出す

まず、「システム上で何が起きるのか」を洗い出します。

例です。

```text
注文する
注文をキャンセルする
支払いをする
商品を出荷する
返品する
返金する
```

このような業務上の出来事を、DDD ではイベントとして捉えることがあります。

ここで大事なのは、画面名や API 名ではなく、**業務で起きていること**として書くことです。

悪い例です。

```text
キャンセルボタンを押す
注文キャンセルAPIを呼ぶ
orders テーブルを更新する
```

良い例です。

```text
顧客が注文をキャンセルする
注文がキャンセル済みになる
返金が必要になる
在庫引当が解除される
```

---

### 2. 業務用語を揃える

次に、関係者が使う言葉を揃えます。

例えば、同じような意味で以下の言葉が混在していると危険です。

```text
注文キャンセル
取消
取り消し
キャンセル済み
無効
返品
返金
```

これらは似ていますが、業務上は別物かもしれません。

| 用語       | 意味                                               |
| ---------- | -------------------------------------------------- |
| キャンセル | 出荷前に注文を取り消すこと                         |
| 返品       | 出荷後・受領後に商品を返すこと                     |
| 返金       | 支払い済みの金額を顧客へ戻すこと                   |
| 無効       | 管理者が不正・重複などを理由に注文を無効化すること |

このように用語を整理すると、実装でもクラス名やメソッド名がぶれにくくなります。

```java
order.cancel();      // キャンセル
order.returnItem();  // 返品
payment.refund();    // 返金
```

> 用語が揃っていないと、コードも設計書もレビューも全部ぶれます。

---

### 3. 判断条件を質問にする

業務ルールは、だいたい「判断条件」として現れます。

設計書を書くときは、以下のような質問をします。

```text
これは誰が実行できますか？
どの状態なら実行できますか？
どの状態なら実行できませんか？
金額・数量・期限などの条件はありますか？
例外ケースはありますか？
処理後に何が変わりますか？
他の業務に影響しますか？
```

注文キャンセルなら、例えばこうです。

```text
Q. 誰がキャンセルできますか？
A. 注文者本人、または管理者のみ。

Q. いつまでキャンセルできますか？
A. 出荷前まで。

Q. 支払い済みの場合はどうしますか？
A. キャンセル後に返金処理を行う。

Q. 在庫はどうなりますか？
A. 引当済み在庫を戻す。

Q. キャンセルできない場合はどうしますか？
A. エラーとして扱い、注文状態は変更しない。
```

この Q&A が、そのまま設計書や実装の材料になります。

---

### 4. 正常系・異常系・境界条件を分ける

業務ルールを整理するときは、最低でも以下に分けます。

| 種類     | 内容                           |
| -------- | ------------------------------ |
| 正常系   | 想定通りに処理できるケース     |
| 異常系   | 処理できないケース             |
| 境界条件 | ちょうど条件の境目になるケース |

例です。

| 種類     | ケース                                     |
| -------- | ------------------------------------------ |
| 正常系   | 未出荷の注文をキャンセルする               |
| 異常系   | 出荷済みの注文をキャンセルしようとする     |
| 異常系   | 存在しない注文 ID でキャンセルしようとする |
| 異常系   | 他人の注文をキャンセルしようとする         |
| 境界条件 | 出荷処理中の注文をキャンセルできるか       |
| 境界条件 | キャンセル済み注文を再度キャンセルできるか |

特に実務では、境界条件が曖昧なまま実装に入ると手戻りが起きやすいです。

## 設計書や実装に必要な情報

設計書や実装で必要な情報は、大きく分けると以下です。

---

### 1. 業務として必要な情報

これは DDD 的に一番大事な情報です。

```text
何をする機能なのか
誰が使うのか
どの業務ルールを満たす必要があるのか
どの状態からどの状態へ変わるのか
できること・できないことは何か
```

注文キャンセルなら、以下が必要です。

```text
注文者本人だけがキャンセルできる
出荷済みの注文はキャンセルできない
キャンセルすると注文ステータスがキャンセル済みになる
支払い済みの場合は返金が必要
在庫引当済みの場合は在庫を戻す
```

これがないと、実装者は「ただ status を更新するだけでよいのか？」を判断できません。

---

### 2. 入力情報

処理を実行するために、外から何を受け取るかです。

```text
注文ID
キャンセル理由
実行ユーザーID
リクエスト日時
```

ここで大事なのは、入力項目だけでなく、**その項目がなぜ必要なのか**も書くことです。

| 項目            | 必須 | 理由                             |
| --------------- | ---- | -------------------------------- |
| 注文 ID         | 必須 | 対象の注文を特定するため         |
| キャンセル理由  | 必須 | 管理・問い合わせ対応で必要なため |
| 実行ユーザー ID | 必須 | 本人確認・権限判定に使うため     |

理由が書かれていると、実装者が入力チェックの重要度を判断しやすくなります。

---

### 3. 出力情報

処理の結果として何を返すかです。

```text
キャンセル後の注文ステータス
キャンセル日時
返金処理の有無
エラー内容
```

API 設計では、成功時だけでなく失敗時の返却も必要です。

```text
成功時
- 注文ID
- 注文ステータス
- キャンセル日時

失敗時
- エラーコード
- エラーメッセージ
```

エラーコードは、画面側がハンドリングできる粒度で決めます。

```text
ORDER_NOT_FOUND
ORDER_ALREADY_SHIPPED
ORDER_ALREADY_CANCELED
PERMISSION_DENIED
```

---

### 4. 状態遷移情報

状態を持つ業務では、状態遷移が必須です。

```text
現在の状態
実行する操作
次の状態
実行可否
```

注文キャンセルなら、以下のように書きます。

| 現在の状態     | キャンセル可否 | キャンセル後の状態 |
| -------------- | -------------- | ------------------ |
| 受付済み       | 可             | キャンセル済み     |
| 出荷準備中     | 可             | キャンセル済み     |
| 出荷済み       | 不可           | 変更なし           |
| キャンセル済み | 不可           | 変更なし           |

この情報があると、Entity に以下のようなメソッドを作りやすくなります。

```java
public void cancel() {
    if (this.status == OrderStatus.SHIPPED) {
        throw new OrderAlreadyShippedException();
    }

    if (this.status == OrderStatus.CANCELED) {
        throw new OrderAlreadyCanceledException();
    }

    this.status = OrderStatus.CANCELED;
}
```

---

### 5. 外部連携情報

外部サービスや他システムと連携する場合は、必ず書きます。

```text
どの外部サービスを呼ぶのか
いつ呼ぶのか
失敗した場合どうするのか
リトライするのか
処理を中断するのか
```

注文キャンセルなら、例えば以下です。

```text
支払い済み注文をキャンセルする場合、決済サービスに返金依頼を送信する。
返金依頼に失敗した場合、注文ステータスは変更しない。
返金成功後、注文をキャンセル済みにする。
```

ここが曖昧だと、実装者は以下を判断できません。

```text
返金に失敗しても注文はキャンセルしてよいのか？
注文をキャンセルしてから返金するのか？
返金してから注文をキャンセルするのか？
リトライは必要なのか？
```

---

### 6. 永続化情報

DB に何を保存するかです。

ただし、DDD / クリーンアーキテクチャでは、いきなり DB から考えるのではなく、

```text
業務上、保存する必要がある情報は何か
```

から考えます。

例です。

```text
注文ステータス
キャンセル理由
キャンセル日時
キャンセル実行者
返金ステータス
```

そのあとで、DB 設計に落とします。

```text
orders.status
orders.cancel_reason
orders.canceled_at
orders.canceled_by
payments.refund_status
```

> DB カラムは、業務情報を保存するための手段です。  
> 最初から DB カラム中心で考えると、業務ルールが抜けやすくなります。

## 情報が足りないときに何を確認するか

実装前に情報が足りないと感じたら、以下を確認します。

---

### 1. 判断条件が足りない

例えば、設計書にこう書かれていたとします。

```text
注文をキャンセルできる。
```

これだけでは実装できません。

確認するべきことは以下です。

```text
どの状態の注文をキャンセルできますか？
出荷済みの場合はキャンセルできますか？
支払い済みの場合はどうしますか？
誰でもキャンセルできますか？
キャンセル理由は必要ですか？
```

設計書には、確認後にこう書きます。

```text
注文は、ステータスが「受付済み」または「出荷準備中」の場合のみキャンセルできる。
出荷済み、キャンセル済みの注文はキャンセルできない。
注文者本人、または管理者のみキャンセルできる。
```

---

### 2. 状態遷移が足りない

設計書にこう書かれていたとします。

```text
キャンセル処理を行う。
```

これだけだと、何をどう変えるのかわかりません。

確認するべきことは以下です。

```text
キャンセル後の注文ステータスは何ですか？
キャンセル日時は保存しますか？
キャンセル者は保存しますか？
返金ステータスは変わりますか？
在庫ステータスは変わりますか？
```

設計書には、確認後にこう書きます。

```text
キャンセル成功時、注文ステータスを「キャンセル済み」に更新する。
キャンセル日時、キャンセル理由、キャンセル実行者を保存する。
在庫引当済みの場合は、在庫引当を解除する。
```

---

### 3. 失敗時の扱いが足りない

実務でよく抜けるのが、失敗時の扱いです。

```text
返金処理に失敗した場合どうするか
DB更新に失敗した場合どうするか
外部APIがタイムアウトした場合どうするか
対象データが存在しない場合どうするか
```

これがないと、実装者が勝手に判断することになります。

確認するべきことは以下です。

```text
処理を中断しますか？
一部だけ成功した状態を許容しますか？
リトライしますか？
エラーとして画面に返しますか？
管理者へ通知しますか？
```

設計書には、確認後にこう書きます。

```text
返金処理に失敗した場合、注文ステータスは変更しない。
エラーコード REFUND_FAILED を返却する。
```

---

### 4. 権限が足りない

誰がその操作をできるのかも重要です。

```text
注文者本人のみ可能なのか
管理者も可能なのか
カスタマーサポート担当者も可能なのか
未ログインでも可能なのか
```

設計書にはこう書きます。

```text
注文キャンセルは、注文者本人または管理者のみ実行できる。
それ以外のユーザーが実行した場合は PERMISSION_DENIED を返す。
```

## 設計から実装までの流れ

DDD とクリーンアーキテクチャを意識する場合、以下の順番で考えると実装しやすいです。

---

### Step 1. 業務イベントを書く

まず、何が起きるのかを書きます。

```text
顧客が注文をキャンセルする。
```

---

### Step 2. 業務ルールを書く

次に、その業務で守るべきルールを書きます。

```text
注文者本人、または管理者のみキャンセルできる。
出荷済みの注文はキャンセルできない。
キャンセル済みの注文は再度キャンセルできない。
支払い済みの場合は返金処理を行う。
在庫引当済みの場合は在庫を戻す。
```

---

### Step 3. 状態遷移を書く

状態が変わる場合は、状態遷移を書きます。

| 現在の状態     | 操作       | 次の状態       | 可否 |
| -------------- | ---------- | -------------- | ---- |
| 受付済み       | キャンセル | キャンセル済み | 可   |
| 出荷準備中     | キャンセル | キャンセル済み | 可   |
| 出荷済み       | キャンセル | -              | 不可 |
| キャンセル済み | キャンセル | -              | 不可 |

---

### Step 4. 入力・出力を書く

```text
入力
- 注文ID
- キャンセル理由
- 実行ユーザーID

出力
- 注文ID
- 注文ステータス
- キャンセル日時
- エラーコード
```

---

### Step 5. ユースケースを書く

ユースケースは、アプリケーションとしての処理の流れです。

```text
1. 注文IDを受け取る
2. 注文情報を取得する
3. 実行ユーザーがキャンセル可能か確認する
4. 注文がキャンセル可能な状態か確認する
5. 支払い済みの場合は返金する
6. 在庫引当済みの場合は在庫を戻す
7. 注文をキャンセル済みにする
8. 結果を返す
```

ここで大事なのは、ユースケースが業務の流れを表し、Controller や DB の詳細を書きすぎないことです。

---

### Step 6. 実装場所に分ける

設計した内容を、クリーンアーキテクチャの各層に分けます。

| 内容                         | 置き場所                 |
| ---------------------------- | ------------------------ |
| 注文がキャンセル可能かの判断 | Entity / Domain Service  |
| 注文キャンセル処理の流れ     | UseCase                  |
| DB から注文を取得・保存      | Repository 実装          |
| API リクエストの受け取り     | Controller               |
| 外部決済サービス呼び出し     | Gateway / Infrastructure |

例です。

```text
Controller
  ↓ リクエストを受け取る
UseCase
  ↓ 処理の流れを制御する
Entity / Domain Service
  ↓ 業務ルールを判断する
Repository / Gateway
  ↓ DBや外部サービスと接続する
```

---

## DDD とクリーンアーキテクチャを併用すると何が便利か

DDD とクリーンアーキテクチャを併用すると、以下が便利になります。

---

### 1. 業務ルールの置き場所が明確になる

DDD だけだと、業務ルールをモデル化する考え方は得られます。  
ただし、実装時にどの層へ置くかは別途考える必要があります。

クリーンアーキテクチャを併用すると、置き場所を整理しやすくなります。

```text
業務ルール
  → Entity / Domain Service

処理の流れ
  → UseCase

DBや外部API
  → Infrastructure

HTTPリクエスト
  → Controller
```

そのため、レビュー時も以下を確認しやすくなります。

```text
この判断ロジックは Controller に置くべきではない
この業務ルールは Entity に寄せた方がよい
この外部API呼び出しは UseCase から interface 経由にした方がよい
```

---

### 2. 技術変更の影響を小さくできる

業務ルールが Controller や DB 処理に混ざっていると、技術変更のたびに業務ロジックまで壊れやすくなります。

例えば、以下の変更があっても、業務ルール自体は本来変わりません。

```text
REST API から GraphQL に変える
MySQL から PostgreSQL に変える
外部決済サービスを変更する
画面を React から別技術に変える
```

クリーンアーキテクチャで分離しておくと、変更の影響範囲を狭められます。

```text
Controller が変わっても UseCase は変えない
DB が変わっても Entity は変えない
外部APIが変わっても業務ルールは変えない
```

---

### 3. テストしやすくなる

業務ルールが Entity や UseCase に分かれていると、DB や画面がなくてもテストできます。

例えば、注文キャンセルのルールなら以下をテストできます。

```text
受付済みの注文はキャンセルできる
出荷済みの注文はキャンセルできない
キャンセル済みの注文は再度キャンセルできない
```

Entity のテスト例です。

```java
@Test
void 出荷済みの注文はキャンセルできない() {
    Order order = new Order(OrderStatus.SHIPPED);

    assertThrows(OrderAlreadyShippedException.class, () -> {
        order.cancel();
    });
}
```

DB や API を通さずに業務ルールを確認できるので、仕様変更にも強くなります。

---

### 4. 設計書とコードの対応がわかりやすくなる

設計書に以下のように書かれているとします。

```text
出荷済みの注文はキャンセルできない。
```

実装では、例えば以下に対応します。

```java
order.cancel();
```

```java
if (this.status == OrderStatus.SHIPPED) {
    throw new OrderAlreadyShippedException();
}
```

このように、設計書の業務ルールとコードが対応していると、後から見た人が理解しやすいです。

> よい設計書は、コードを読む前に「どこに何が実装されるべきか」が想像できます。

---

## まとめ

DDD とクリーンアーキテクチャを使う場合、基本設計書では以下を意識します。

```text
1. 画面やDBではなく、業務として何が起きるかから書く
2. 業務ルールと技術処理を分けて書く
3. 状態が変わる条件を必ず書く
4. 正常系・異常系・境界条件を書く
5. 入力・出力・権限・外部連携・保存情報を書く
6. 不明点は判断条件・状態遷移・失敗時の扱いとして確認する
7. 設計した業務ルールを Entity / UseCase / Infrastructure に分けて実装する
```

一番大事なのは、以下です。

> DDD は、業務ルールを正しく見つけるための考え方。  
> クリーンアーキテクチャは、その業務ルールを技術都合から守るための構造。

設計書を書くときは、単に「何が大事か」を書くだけではなく、

```text
何を確認するのか
どう書くのか
どこに実装するのか
```

まで落とすと、実装者が自走しやすくなります。
