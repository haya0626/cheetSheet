# DDD × クリーンアーキテクチャ 設計・実装ガイド

> DDD とクリーンアーキテクチャは、どちらも「コードをきれいに分けるため」だけの考え方ではありません。
> 本質は、業務ルールを正しく理解し、変更に強い形で実装へ落とすことです。

## DDD とクリーンアーキテクチャの役割

DDD とクリーンアーキテクチャは担当する関心が少し異なる

| 考え方                 | 主な関心                                 | 簡単にいうと                             |
| ---------------------- | ---------------------------------------- | ---------------------------------------- |
| DDD                    | 業務をどう理解してモデル化するか         | 業務ルールをコードで表現するための考え方 |
| クリーンアーキテクチャ | コードをどう分離して依存関係を整理するか | 業務ルールを技術都合から守るための構造   |

DDD は、

> この業務では、何が重要な概念なのか？  
> どんなルールで状態が変わるのか？

を考えるためのもの

クリーンアーキテクチャは、

> その業務ルールを、DB・画面・API・外部サービスに引っ張られない形でどこに置くか？

を考えるためのもの

つまり、ざっくりいうと以下

```text
DDD
  ↓
業務ルールを見つける・整理する
```

```text
クリーンアーキテクチャ
  ↓
整理した業務ルールを守れる場所に配置する
```

## 作業者に求められる視点

DDD とクリーンアーキテクチャを前提に作業する場合、作業者には以下の視点が求められる

### 業務の言葉で理解する

実装都合の言葉ではなく、業務上の意味で処理を理解する必要があります。

悪い例：

- 注文ステータスを CANCELED に更新する

良い例：

- 未出荷の注文はキャンセルできる
- キャンセル後、注文はキャンセル済みになる
- 出荷済みの注文はキャンセルできない

このように、処理を「業務上の状態変化」として捉えることが重要

---

### 業務ルールと技術処理を分ける

業務ルールと技術処理が混ざると、どこが仕様で、どこが実装都合なのか分からなくなる

| 種類       | 内容                                                           |
| ---------- | -------------------------------------------------------------- |
| 業務ルール | キャンセルできる条件、金額計算、状態遷移、制約など             |
| 技術処理   | API、DB、SQL、HTTP、トランザクション、外部サービス呼び出しなど |

業務ルールは Entity や Domain Service に寄せ、処理の流れは UseCase に置き、DB や外部 API との接続は Infrastructure に分けることで、責務が明確になる

## 認識ブレを防ぐための解決策・運用方法

設計思想を守るには、「DDD を理解してください」と伝えるだけでは不十分

現場では、作業者が迷わないための資料・テンプレート・レビュー観点を仕組み化することが重要

---

### ユビキタス言語の用語集を作る

関係者が同じ言葉を同じ意味で使えるように、業務用語を整理する

例：

| 用語       | 意味                           |
| ---------- | ------------------------------ |
| キャンセル | 出荷前に注文を取り消すこと     |
| 返品       | 出荷後、商品を返却すること     |
| 返金       | 支払い済み金額を顧客へ戻すこと |

用語が曖昧なままだと、設計書、コード、レビュー、テスト観点がすべてブレます

---

### 状態遷移表を作る

状態を持つ業務では、状態遷移表を作成します。

| 現在の状態     | 操作       | 次の状態       | 可否 | 理由                       |
| -------------- | ---------- | -------------- | ---- | -------------------------- |
| 受付済み       | キャンセル | キャンセル済み | 可   | 出荷前のため               |
| 出荷準備中     | キャンセル | キャンセル済み | 可   | 出荷前のため               |
| 出荷済み       | キャンセル | -              | 不可 | すでに出荷済みのため       |
| キャンセル済み | キャンセル | -              | 不可 | すでにキャンセル済みのため |

状態遷移表があると、実装者は判断ロジックをどこに書くべきか判断しやすくなる

## 設計に必要な情報

設計するためには、最低限以下の情報が必要

- 何をする機能なのか
- 誰が使うのか
- どの業務ルールを満たす必要があるのか
- 何ができるのか
- 何ができないのか
- 処理後に何が変わるのか

### 判断条件

- 誰が実行できるのか
- どの状態なら実行できるのか
- どの状態なら実行できないのか
- 金額、数量、期限などの条件はあるか
- 例外ケースはあるか

### 外部連携情報

- どの外部サービスを呼ぶのか
- いつ呼ぶのか
- 失敗した場合どうするのか
- リトライするのか
- 処理を中断するのか

これらがないと、設計者は「業務として何が正しいのか」を判断できない

## 設計時に追加で確認すべき情報

DDD × クリーンアーキテクチャを前提にする場合、通常の設計情報に加えて、以下も確認する必要がある

- この業務で中心になる概念は何か
- その概念は Entity として扱うべきか
- 値だけで表現できるものは Value Object にできるか
- 業務ルールはどのドメインモデルに持たせるか
- ひとつの処理の流れはどの UseCase として表現するか
- DB や外部 API に依存しない形で業務ルールを書けるか
- Controller や Repository に業務判断が漏れていないか

設計資料上はきれいでも、実装時に業務ルールの置き場所がばらつき、結果としてクリーンアーキテクチャの構造が崩れやすくなる

## DDD とクリーンアーキテクチャを併用するメリット

DDD とクリーンアーキテクチャを併用すると、以下のメリットがある

### 業務ルールの置き場所が明確になる

DDD で業務ルールを整理し、クリーンアーキテクチャで実装場所を分けることで、どこに何を書くべきかが明確になる

```text
業務ルール → Entity / Domain Service
処理の流れ → UseCase
DB・外部API → Infrastructure
画面・API入口 → Controller
```

---

### 変更に強くなる

業務ルールが Controller や DB 処理に混ざっていると、画面変更や DB 変更の影響を受けやすくなる

クリーンアーキテクチャで分離しておくことで、技術変更が発生しても、業務ルールへの影響を小さくできる

---

### レビューしやすくなる

責務が分かれているため、レビュー観点が明確になる

- 業務ルールが適切な場所にあるか
- UseCase に技術詳細を書きすぎていないか
- Controller に業務判断が入っていないか
- Repository に業務ルールが漏れていないか

このように確認できるため、設計思想をチームで維持しやすくなる
