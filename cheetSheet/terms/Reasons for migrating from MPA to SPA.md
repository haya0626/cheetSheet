# MPA（JSP）から SPA へ移行した背景と理由

## 1. 従来の MPA（JSP）構成とは

MPA では、以下のような流れで画面が構築される。

- ユーザー操作
- HTTP リクエスト送信
- サーバー側で処理
- JSP をレンダリング
- HTML を再送信
- 画面を再描画（ページ遷移）

### 特徴

- 画面遷移ごとに `HTML 全体を再生成`
- 表示ロジックと業務ロジックが `サーバー側に集中`
- フロントエンドは基本的に「画面テンプレート」

## 2. MPA 開発で顕在化した問題

### 2.1 ユーザー体験（UX）の限界

- ページ遷移のたびに画面が白くなる
- 部分的な更新でも `画面全体の再読み込み`
- 操作のレスポンスが遅く感じられる

特に、入力補助・リアルタイム検索・動的 UI が求められるようになるにつれ、MPA の体験は時代に合わなくなっていった。

### 2.2 画面ロジックの肥大化

JSP では以下が混在しがちになる。

- HTML
- Java の制御構文
- 業務ロジックの一部
- 画面制御フラグ

結果として：

- JSP が **巨大化・スパゲッティ化**
- 変更影響範囲が把握しづらい
- 新人・別担当者が読むのが困難

### 2.3 フロントとバックエンドの責務が曖昧

- 画面制御はどこまでサーバーでやるのか？
- JavaScript は補助か、処理の一部か？
- validation はどこで行うのか？

役割分担が設計者ごとに異なり、`チーム内の設計が統一できない` という問題が発生した。

### 2.4 開発の分業が難しい

MPA 構成では：

- 画面と業務が密結合
- フロントとバックエンドを `別チームで並行開発しづらい`
- UI 改善のたびにサーバー改修が必要

これにより、開発スピードが上がらなくなった。

## 3. Web 利用環境の変化

### 3.1 リッチな UI 需要の増加

- Google Maps
- SNS
- 業務 Web アプリの高度化

「画面はただ表示できればよい」という時代から、`操作性・即時性が重要` な時代に変化した。

### 3.2 JavaScript とブラウザ性能の進化

- JavaScript エンジンの高速化
- DOM 操作性能の向上
- Web API の充実

ブラウザ側で複雑な処理を行える土台が整った。

## 4. SPA が選ばれるようになった理由

### 4.1 SPA の基本構成

- 初回に 1 ページだけ読み込む
- 画面遷移は JavaScript で制御
- API を通じてデータのみ取得
- DOM を部分更新

### 4.2 SPA が解決した課題

#### UX の改善

- ページ遷移なし
- 高速な画面切り替え
- 入力中の即時反映

#### 責務分離の明確化

- フロント：表示・操作・状態管理
- バックエンド：業務ロジック・データ提供（API）

#### 開発体制の柔軟化

- フロントエンド／バックエンドの分業が可能
- API 契約を決めれば並行開発できる

## 5. MPA → SPA 移行の本質的な理由

単なる「技術トレンド」ではなく、

- `ユーザー体験の要求が上がった`
- `システム規模が大きくなった`
- `開発効率・保守性が限界に達した`

これらに対する **必然的な進化** として SPA が選択されている。

## 6. まとめ

| 観点     | MPA（JSP）   | SPA          |
| -------- | ------------ | ------------ |
| 画面更新 | ページ全体   | 部分更新     |
| UX       | 遷移が重い   | 高速・滑らか |
| 責務分離 | 曖昧         | 明確         |
| 開発分業 | 困難         | 容易         |
| 保守性   | 低下しやすい | 向上         |

現在の SPA 採用は、「新しい技術を使いたいから」ではなく、 **システム開発の課題に対する合理的な回答**である。
