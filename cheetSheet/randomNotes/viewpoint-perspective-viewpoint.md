# 視座 / 視野 / 視点

- 視野 = 問題を発見する力
- 視点 = 問題を評価する力
- 視座 = 問題の“意味・優先度・文脈”を決める力

## 視点の多さ : どれだけ多面的に評価・判断できるか

## 視点とは

対象を評価するときに用いる「観点の数」と「バランス感」

- 複数の観点を同時に扱える
- トレードオフを理解している
- 状況に応じて観点の重み付けを変えられる

具体的にできること

- 短期最適と長期最適を切り分ける
- 技術的に正しいだけでなく、ビジネス的に正しい判断ができる

## 視野の広さ : どこまでを“影響範囲”として認識できているか

### 視野とは

自分が対象として捉えている範囲の広さ

- 自分 → チーム → プロダクト → ユーザー → ビジネスと広がっていく

具体的にできること

- 変更の影響範囲を事前に想像できる
- 他職種（デザイナー・バックエンド）との関係を理解できる
- 「自分のタスク」ではなく「全体の流れ」で考えられる

## 視座の高さ : どのレイヤーの目的で意思決定しているか

### 視座とは

思考を置く立場の高さ

- 実装者視点
- チーム視点
- 事業視点
- 経営・社会視点

具体的にできること

- 判断理由を上位目的に紐づけられる
- 局所最適ではなく全体最適を選べる
- 技術選定に「意図」がある

## エンジニアでの具体例

### ① 視点の多さ（UI 改善のケース）

例：ボタンのデザインを変更する

視点が少ない人

- 見た目が良くなったかだけで判断
- 「デザイン的にこっちがいい」で終わる

視点が多い人（複数観点で評価）

- UX：押しやすいか？誤操作しないか？
- パフォーマンス：描画コスト増えてないか？
- アクセシビリティ：コントラスト比 OK？キーボード操作可能？
- 一貫性：他画面と UI ルール揃ってるか？
- 実装コスト：コンポーネント使い回せるか？
- ビジネス：CVR 上がる可能性あるか？

違い

- 視点が少ない → 局所的に“良さそう”
- 視点が多い → 「総合的に最適」を選べる

## ② 視野の広さ（機能実装のケース）

例：検索機能の UI を変更する

視野が狭い人

- 自分の担当コンポーネントだけ見る
- 「動いたから OK」で終わる

視野が広い人（最初から以下を考える）

- API のレスポンス形式と整合性
- 検索結果が他ページにどう影響するか
- ローディング・エラーハンドリングの UX
- モバイル・PC での挙動差
- 他の検索機能との一貫性
- 将来的な拡張（フィルター追加など）

違い

- 視野が狭い → バグや手戻りが多い
- 視野が広い → 事前にリスク潰せる

## ③ 視座の高さ（作業者視点での具体例）

例：フォーム画面を実装する

（入力フォーム＋バリデーション＋送信処理）

視座が低い（作業者としての最小単位）

思考

- 「とりあえず動けば OK」
- 「仕様通りに作る」
- 「エラー出なければ完成」

行動

- その場しのぎのバリデーションを書く
- コンポーネントが画面ごとにバラバラ
- 命名や構造に一貫性がない

コードの特徴

- コピペが多い
- 再利用されない
- 将来の変更に弱い

視座が一段上（チーム視点を持った作業者）

思考

- 「他の人も触るコード」
- 「レビューしやすいか？」
- 「他画面でも使えるか？」

行動

- バリデーションロジックを共通化
- コンポーネントを分割・再利用可能にする
- 命名や設計をチームルールに合わせる

コードの特徴

- 再利用できる
- 読みやすい
- 修正コストが低い
